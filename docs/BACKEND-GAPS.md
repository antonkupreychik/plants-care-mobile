# Backend gaps — найдено при разработке мобилки

> Живой лог: что мобильному клиенту нужно от API, но в контракте этого нет или
> оно неудобно. Заполняется по ходу фич. Бэкенд дорабатывается отдельно
> (владелец), мобилка пока ставит заглушку и помечает `TODO(gap #N)`.
>
> Это дополняет `design_handoff_plantcare/api-contract.md` §12 (исходный список
> от дизайна) — здесь фиксируем то, что реально упёрлись при имплементации.
>
> Статусы: 🔴 нет в API · 🟡 есть, но неудобно/неполно · 🟢 закрыто бэкендом.

## Легенда полей
`ID` — локальный номер · `Экран` · `Нужно` · `Сейчас в API` · `Предложение` · `Статус`

## G1 · Health Score растения 🔴
- **Экран:** 01 Home (мини-кольцо на карточке), 02 Plant card (бейдж).
- **Нужно:** числовой score 0–100 на растение (для кольца) + опц. факторы.
- **Сейчас:** нет. `PlantDto` без health-поля; `/plants/{id}/health` отсутствует.
- **Предложение:** поле `healthScore: int?` в `PlantDto` (дёшево для списка) ИЛИ
  `GET /api/v1/plants/{id}/health → {score, factors[]}`. Совпадает с api-contract §12.4 (#138).
- **Заглушка мобилки:** кольцо health не рисуем / рисуем нейтральным, пока поля нет.

## G2 · Mood / voice line растения 🔴
- **Экран:** 01 Home (подпись-настроение), 02 Plant card (speech bubble).
- **Нужно:** `status` (`happy|thirsty|sick`) и/или текст реплики от первого лица.
- **Сейчас:** нет. Бренд-фича, текст пока генерится на клиенте.
- **Предложение:** `status`/`moodLabel` в `PlantDto`, `voiceLine` в `TaskDto`/`PlantDto`
  (api-contract §12.3). Решить: клиент или бэк генерит реплики.
- **Заглушка мобилки:** генерим на клиенте по `taskType` + просрочке.

## G3 · Алерт «проблемное растение» на Home 🔴
- **Экран:** 01 Home (баннер сверху).
- **Нужно:** признак, что у растения проблема (для баннера) + ссылка на растение.
- **Сейчас:** нет.
- **Предложение:** флаг в `/today` или `/me` (`attentionPlantId?`), либо вывести из G1 (health < порога).
- **Заглушка:** баннер не показываем, пока нет сигнала.

## G4 · Виджет погоды на Home 🔴
- **Экран:** 01 Home (микро-строка погоды).
- **Сейчас:** нет. api-contract §12.8 (#69) — `GET /weather/snapshot`.
- **Заглушка:** строку погоды скрываем.

## G5 · Счётчики пользователя (header Home) 🔴
- **Экран:** 01 Home (приветствие, badge уведомлений).
- **Нужно:** `{plantsTotal, tasksToday, notificationsUnread}` + имя/аватар.
- **Сейчас:** нет `/me`. api-contract §12.1.
- **Заглушка:** `tasksToday` берём из длины `/today`; имя — заглушка; badge не рисуем.

## G6 · Привязка задачи/растения к виду (иллюстрация) 🟡
- **Экран:** 01 Home, 03 Today — нужно выбрать SVG-иллюстрацию (Monstera/Fern/...).
- **Нужно:** по `TaskDto`/`PlantDto` понять вид, чтобы выбрать иллюстрацию.
- **Сейчас:** `PlantDto.speciesId/speciesName` есть; `TaskDto` (в /today) — только `plantName`,
  без `speciesId`. На Today-задаче не из чего выбрать иллюстрацию.
- **Предложение:** добавить `speciesId` (или ключ иллюстрации) в `TaskDto`.
- **Заглушка:** на Today дефолтная иллюстрация; на Home — по `speciesName` маппингом.

## G7 · `taskType` (today/calendar) ≠ `type` (care-events) 🟡
- **Везде, где выполняем уход.**
- `/today`,`/calendar` отдают `WATERING/MISTING/FERTILIZING/SOIL_CHECK`, а `POST /care-events`
  принимает `WATER/SPRAY/FERTILIZE`. Маппинг делаем на клиенте (domain-слой).
- **Предложение бэку:** согласовать enum’ы или отдавать оба. Низкий приоритет (маппинг дешёвый).

## G8 · Рантайм `/v3/api-docs` не документирует заголовки/параметры 🟡
- **Тулинг (кодген).** springdoc на dev-хосте не видит `X-User-Id`/`X-Chat-Id` и query-параметры
  (`locationId/offset/limit`) — PoC резолвит заголовки через `UserApiResolver` мимо аннотаций.
  `/calendar` отдаёт пустую схему (springdoc не сериализовал `Map<String,List<TaskDto>>`).
- **Следствие:** кодген идёт из статической `api/openapi/`, а не из рантайма (см. memory
  `openapi-codegen-source`).
- **Предложение бэку:** добавить `@Parameter` для заголовков/квери и именованную
  `CalendarResponse`-обёртку, чтобы `/v3/api-docs` стал пригоден как источник.

## G9 · `GET /plants` отдаёт 500 для user 1 (dev) 🟢
- **Экран:** 01 Home (сетка растений).
- **Симптом (был):** `GET /api/v1/plants` с `X-User-Id: 1` на dev-хосте → `500 INTERNAL_ERROR`
  (проверено curl 2026-05-27). `/locations` (200) и `/today` (404) для тех же id отвечают.
- **Следствие (было):** на Home секция растений уходит в ErrorState. Клиент обрабатывает
  корректно (это не баг мобилки), но протестировать «живой» сад на dev было нельзя.
- **Закрыто (2026-05-27):** бэкенд починен. `GET /api/v1/plants` с `X-User-Id: 1` теперь
  отдаёт `200` с реальными данными (растения user 1, напр. «Монстера» id 7, 25). Проверено
  curl с/без `Accept`, а также с `Accept: application/json` — все варианты `200`. На Home
  можно тестировать «живой» сад. (`/today` для user 1 всё ещё `404` — отдельный вопрос.)

## G10 · `GET /plants/{id}/history` отдаёт 500 (dev) 🟢
- **Экран:** 02 Карточка растения (лента истории ухода).
- **Симптом (был):** `GET /api/v1/plants/{id}/history` с валидным `X-Chat-Id: 764602851`
  → `500 INTERNAL_ERROR`. Воспроизводилось для всех растений и любых параметров.
  Проверено curl 2026-05-27.
- **Следствие (было):** секция «История ухода» на экране 02 уходила в ErrorState
  (деталь и стрик на той же карточке работали).
- **Закрыто (2026-05-27):** бэкенд починен. `GET /plants/{id}/history` теперь отдаёт
  `200` с реальными событиями (`items[]` с `type` WATER/SPRAY, `performedAt`, `onTime`).
  Форма совпадает с маппером `care_history_mapper`. Секция истории на экране 02
  наполняется.

## G11 · `/calendar` без признака выполнения задач 🔴
- **Экран:** 11 График (недельный календарь).
- **Нужно:** по задаче в `/calendar` понимать, выполнена ли она — для прогресса дня
  `done/count` (полоса в дизайне экрана 11) и состояния «готово»/«просрочено».
- **Сейчас:** `GET /api/v1/calendar` отдаёт только **запланированные** задачи (`TaskDto`:
  `scheduleId/plantId/plantName/taskType/nextDueAt`). Признака выполнения нет — `done`
  посчитать нельзя. Та же `TaskDto`, что в `/today`.
- **Предложение:** добавить в `TaskDto` (для calendar) флаг/время выполнения, например
  `doneAt: string?` (или `status: PENDING|DONE`), либо отдельную карту выполненных
  `care-events` на диапазон, чтобы клиент сматчил по `scheduleId`+дате.
- **Заглушка мобилки:** прогресс-бар `done/count` из дизайна **не рисуем**; в строке дня
  показываем счётчик задач (`scheduleDayTasksCount`) или «Свободно». «Просрочено»
  выводим клиентом по `nextDueAt < now` (раскрытый сегодняшний день). «Готово» нигде.

## G12 · Экспорт календаря `.ics` отсутствует 🔴
- **Экран:** 11 График (карточка «Подписаться в календаре»).
- **Нужно:** `GET /api/v1/calendar.ics` — подписка в Google/Apple Calendar
  (api-contract §11 «Хотелки», #79).
- **Сейчас:** эндпоинта нет.
- **Предложение:** `GET /api/v1/calendar.ics` (scope chat), отдающий VCALENDAR с задачами
  ухода; стабильный URL для подписки.
- **Заглушка мобилки:** карточка `.ics` (`ScheduleIcsCard`) отрисована, но **инертна** —
  тап → snackbar «Скоро» (`comingSoon`). Включим, когда появится эндпоинт.

> **Клиентская ловушка (не backend-гэп), экран 11.** Спека объявляет query `from`/`to`
> у `/calendar` как `type: string, format: date` (`YYYY-MM-DD`) — это **корректно**, бэкенд
> на `YYYY-MM-DD` отвечает `200`. Но `swagger_parser`/`retrofit` генерят их как Dart
> `DateTime` и сериализуют через `toIso8601String()` → уходит `2026-05-20T00:00:00.000`,
> на что бэк отдаёт `500`. Воркэраунд — рантайм-интерсептор `DateQueryInterceptor`
> (`core/network`, opt-in через `dateOnlyQueryExtra({'from','to'})`), усекающий значение
> до даты. Сгенерированный код НЕ правился. Снять, когда кодген начнёт сериализовать
> `format: date` как date-only.

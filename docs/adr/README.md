# Mobile ADR (MADR)

Архитектурные решения по **мобильному клиенту** PlantCare (Flutter).

Нумерация **отдельная** от backend (`ADR-001…010`), префикс `MADR-`, чтобы не путать
два потока решений в одном продукте. Backend-ADR живут в репозитории `plants-care` и
остаются источником правды по доменной модели, шедулеру и контракту API.

## Статусы

- **Proposed** — предложено архитектором, ждёт подтверждения владельца.
- **Accepted** — подтверждено владельцем, действует.
- **Superseded by MADR-NNN** — заменено другим решением.

> Перевод в **Accepted** делает владелец проекта (не архитектор, не фича-агент). Гейт из
> FLUTTER.md: фича-агентов не запускать, пока ключевые MADR не подтверждены и каркас не собран.
>
> **2026-05-27 — владелец прошёл по трём развилкам из SPRINT-1 §0.5 и зафиксировал выбор.**
> По двум (MADR-007, MADR-012) решение владельца **разошлось** с рекомендацией архитектора —
> тела этих MADR переписаны под принятое решение, рекомендация сохранена как отвергнутая
> альтернатива. Остальные MADR пока **Proposed** (владелец их явно не ратифицировал).

**Три развилки из SPRINT-1 §0.5 — решения владельца (2026-05-27):**

- **MADR-007** — ✅ **кодген dart-dio из OpenAPI** (рекомендация архитектора была: ручные DTO).
- **MADR-012** — ✅ **ru-only сейчас**, но все строки через `AppLocalizations` (рекомендация была: ru+en).
- **MADR-013** — ✅ **GitHub Actions сейчас → Codemagic к релизу** (совпало с рекомендацией).

> Остальные MADR (001–006, 008–011, 014) остаются **Proposed**. Их можно ратифицировать

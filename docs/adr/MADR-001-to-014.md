## MADR-002: Архитектура слоёв — Clean Architecture + MVVM

**Статус:** Proposed

**Контекст.** Владелец — backend-разработчик (Spring Boot), для него понятна расслойка
controller → service → repository. Нужна архитектура, которая ложится на этот опыт и
переживёт рост до 24 экранов и backend-gaps.

**Решение.** Три слоя на фичу:
- **data** — `*Dto` (зеркало JSON backend), `*RepositoryImpl`, `*RemoteDataSource`
  (dio), `*LocalDataSource` (drift). Маппинг DTO ↔ domain — явный, в `mappers/`.
- **domain** — иммутабельные модели (freezed), интерфейсы репозиториев, use cases.
  **Чистый Dart**: ни одного `import 'package:flutter/...'` или Riverpod.
- **presentation** — Riverpod Notifier/AsyncNotifier (роль ViewModel) + экраны/виджеты.

Зависимости строго вниз: presentation → domain ← data. Domain ничего не знает о data и
presentation. (Аналогия: domain = ваши Spring `@Service`-интерфейсы и сущности, data =
JPA-репозитории + DTO, presentation = контроллеры + DTO ответа.)

**Причины.** Тестируемость (domain юнит-тестится без Flutter), замена источника без
переписывания UI (PoC-заголовки → JWT, REST → offline-first), и прямой перенос ментальной
модели владельца. Трейд-офф — больше файлов и ручной маппинг DTO↔domain; для соло это
кажется лишним, но именно маппинг спасает от ловушки `taskType` (WATERING) ≠ `type`
(WATER) из api-contract: разные DTO мапятся в один domain-enum.

**Последствия.** На каждую сущность 2 модели (Dto + domain) и маппер. Для тривиальных
справочников (`care-types`) допустим один класс без отдельного domain, если он не несёт
логики — но это исключение, фиксируется в ревью.

## MADR-003: Структура проекта — feature-first

**Статус:** Proposed

**Контекст.** 24 экрана, 6 секций, много backend-gaps (фичи добавляются волнами).
Layer-first (`lib/data`, `lib/domain`, `lib/presentation`) на таком объёме превращается
в свалку: чтобы тронуть одну фичу, лезешь в три далёких папки.

**Решение.** Feature-first:

```
lib/
├── core/                 # общее: network, router, theme, storage, errors, l10n, env
├── features/<feature>/{data,domain,presentation}/
├── main_dev.dart
└── main_prod.dart
```

Правила: фича **не** импортит `presentation/` другой фичи. Шарится только через `core/`
или через domain-интерфейс (если фиче B нужны данные фичи A — B зависит от domain-репозитория
A, не от его экранов). Sprint 1 фичи: `home`, `plant_card`, `care_event`, `auth`, плюс
общие справочники (`species`, `locations`) — в `core/` или в `catalog`-фиче (решить при
нарезке Sprint 2).

**Причины.** Локальность изменений (вся фича в одной папке = легко дать агенту), границы
видны в дереве, удаление фичи = удаление папки. Трейд-офф — соблазн дублировать общий код;
лечится дисциплиной «общее → core».

**Последствия.** Нужен явный список «что в core»: network, router, theme, storage (drift),
errors, l10n, env, общие виджеты (`core/widgets/`), Clock-провайдер.

## MADR-004: State management — Riverpod 3 + codegen

**Статус:** Proposed

**Контекст.** Нужен server-state (кеш + инвалидация после мутаций, как TanStack Query в
RN-хэндоффе) и UI-state (тема, фильтры, активная локация). Альтернативы: Bloc (строже, но
больше boilerplate — для соло избыточно), Provider (проще, но перерастётся на 24 экранах).

**Решение.** Riverpod 3 с генерацией (`@riverpod`).
- Асинхронные данные (API/БД) — `AsyncNotifier` / функциональные `@riverpod`-провайдеры,
  возвращающие `Future`/`Stream`. Loading/error — через `AsyncValue` в UI.
- UI-состояние — `Notifier`.
- **DI отдельным пакетом (get_it) не вводим** — провайдеры и есть граф зависимостей.
- **Инвалидация после `POST /care-events`** (перенос ключей из README §5): `ref.invalidate`
  провайдеров `today`, `plant(id)`, `plantHistory(id)`, `streak(id)`, `calendar`.
- `ref.watch` — в build; `ref.read` — в колбэках.

**Причины.** Compile-time безопасность провайдеров, нет зависимости от BuildContext,
тестируется через `ProviderContainer` с `overrides`, минимум boilerplate с codegen. Для
соло-разработчика — лучший баланс «строгость / скорость».

**Последствия.** Все провайдеры — через build_runner (codegen). UI читает `AsyncValue` и
рисует skeleton/error из коробки. Нет ручного управления подписками.

## MADR-005: Навигация — go_router

**Статус:** Proposed

**Контекст.** 24 экрана: bottom-nav с табами (Home / График / Каталог / Профиль),
push-экраны (карточка растения, мастер из 4 шагов), модальные sheet'ы (полив/опрыск/подкормка),
auth-флоу, deep links под Telegram magic-link (Phase 3) и snooze из push (Phase 5).

**Решение.** `go_router` 17.x.
- **StatefulShellRoute** для bottom-nav (сохраняет состояние каждого таба).
- Push-экраны (`/plant/:id`, мастер) — обычные роуты поверх shell.
- Sheet'ы ухода — **не роуты**, а `showModalBottomSheet` из колбэка (это transient UI,
  не навигационное состояние). Это согласуется со SPRINT-1 §7.
- Auth-флоу (07/08/09) — отдельная ветка вне shell; redirect-guard по auth-провайдеру.
- Типобезопасные роуты через `@TypedGoRoute` (codegen) — для `/plant/:id` и мастера.
- deep links (`plantcate://…`) закладываем структурой, активируем в Phase 3/5.

Черновик дерева:
```
/auth/welcome           (07)   /auth/telegram (08)   /auth/welcome-back (09)
ShellRoute [tabs]:
  /home                 (01)  └─ /plant/:id          (02)
  /calendar             (11)
  /catalog              (12)  └─ /species/:id
  /profile              (13)
/add-plant/step1..step4 (04a/04b/04/04c)  — push поверх shell
(модалки: water/spray/fert — showModalBottomSheet, не роуты)
```

**Причины.** Декларативная, нативные deep links, redirect-guards под auth, типобезопасность.
Альтернатива (Navigator 1.0) не тянет deep links и guards без боли.

**Последствия.** Один `GoRouter`-провайдер в `core/router/`. refreshListenable на auth-стейт.
Sheet'ы не в роутере — значит back-кнопка их не закрывает через router (закрывает Navigator).

## MADR-006: Сетевой слой — dio, две «головы» заголовков, ApiError

**Статус:** Proposed

**Контекст.** PoC-auth backend: **два разных заголовка** в зависимости от эндпоинта —
`X-User-Id` (/plants, /locations) и `X-Chat-Id` (/today, /calendar, /care-events,
/plants/{id}/history, /stats/streak); справочники (/species, /care-types, /health) — без
заголовков. Единый формат ошибок `{error:{code,message,details[]}}`. Идемпотентность
care-events через `clientId`.

**Решение.** Один `Dio`-инстанс в `core/network/`, base URL из `--dart-define API_URL`.
Интерсепторы:
1. **AuthHeaderInterceptor** — выбирает заголовок по пути запроса. Реализация — через
   **per-request extra-флаг** (`options.extra['auth'] = AuthScope.user | AuthScope.chat | AuthScope.none`),
   который проставляет data source, а интерсептор читает и подставляет нужный заголовок.
   Это надёжнее, чем матчить URL регуляркой (пути меняются), и переживёт переход на JWT:
   тогда интерсептор просто ставит `Authorization: Bearer` независимо от scope (см. MADR-008).
2. **ErrorInterceptor** — мапит `DioException` + тело `{error:{...}}` в `ApiError`
   (sealed, freezed): `validation(details)`, `accessDenied`, `notFound`, `conflict`,
   `locationNotEmpty`, `network`, `unknown`. Маппинг по `error.code`.
3. `dio_smart_retry` — ретраи только на сетевые/5xx (идемпотентные GET; POST care-events
   ретраить безопасно из-за `clientId`).

**Care-events ловушки** (фиксируются как тест в data-слое care_event):
- `clientId = Uuid().v4()` **генерится один раз** на тап (хранится в state, не на каждый
  build) — иначе ретрай создаст новый UUID и продублирует полив.
- маппер `taskType (WATERING) → type (WATER)` живёт в domain (см. MADR-002).
- `/calendar` — клиент сам не запрашивает > 60 дней (валидация перед запросом).

**Причины.** Per-request scope-флаг развязывает «какой заголовок» от «какой URL» и делает
переход на JWT заменой одного интерсептора, а не переписыванием всех data sources.

**Последствия.** Data source обязан указывать `AuthScope` для каждого запроса (дефолт —
`none` для безопасности: забыл — упадёт на backend с понятной 401/403, а не утечёт чужой заголовок).

## MADR-007: API-клиент — кодген dart-dio из OpenAPI

**Статус:** Accepted (2026-05-27, владельцем) · решение разошлось с рекомендацией архитектора

**Контекст.** Развилка из SPRINT-1 §0.5/§3: генерить dart-dio клиент из `openapi.yaml`
(swagger_parser / openapi-generator) или писать DTO руками на freezed. OpenAPI разбит
через `$ref` (нужен bundle), а DTO в любом случае мапятся в domain-модели (MADR-002).

**Решение.** **Кодген dart-dio клиента из OpenAPI.** Инструмент — `swagger_parser`
(собирает dio-клиент + DTO; bundle `$ref` делаем шагом сборки). DTO — сгенерированные,
а не написанные руками.

**Причины.**
- Контракт — единый источник правды на backend; сгенерированные DTO не дают полям и типам
  «разъехаться» с реальным API при правках на стороне сервера (перегенерировал — увидел дифф).
- Меньше рутинной ручной работы на ~12 эндпоинтах сейчас и на backend-gaps, которые приходят
  волнами: новый эндпоинт = перегенерация, а не ручная переписка DTO.
- Снимает класс ошибок «опечатка в имени поля / забыл nullable».

**Как закрываем минусы кодгена** (то, ради чего архитектор предлагал ручные DTO):
- **«Грязный» PoC-контракт** (`taskType ≠ type`, `PUT` с PATCH-семантикой, `CalendarResponse`
  как `Map<String, List<...>>`, заголовки-«головы») — генератор отдаёт это as-is. Поэтому
  **слой маппинга DTO → domain (MADR-002) остаётся обязательным и пишется руками**: именно там
  ловушки нормализуются (разные DTO → один domain-enum). Маппер — точка контроля, не клиент.
- **`$ref`-разбивка** — добавляем шаг bundle (swagger-cli/redocly) перед генерацией в CI/скрипте.
- **Сгенерированный код не правим руками** — только через перегенерацию; правки контракта идут
  на backend. Сгенерированную папку коммитим (воспроизводимость) и помечаем в ревью как «не трогать».

**Последствия.** В `pubspec` (dev) входит `swagger_parser` (версию зафиксировать при сборке
каркаса — сверить на pub.dev). `openapi.yaml` скачивается в `api/`, бандлится, генерация —
скриптом (`tool/gen_api.sh`) + в CI. Сгенерированные DTO/клиент — в `core/api/generated/` (или
`features/*/data/dto/` по результату генерации). Round-trip-тесты на сгенерированные DTO не
нужны (за корректность отвечает генератор) — тестами покрываем **мапперы** DTO ↔ domain.

> **Отвергнутая альтернатива (рекомендация архитектора): ручные DTO на freezed.** Аргумент —
> полный контроль над маппингом «грязного» PoC-контракта и меньше движущихся частей в
> build_runner. Владелец выбрал кодген ради синхронизации с контрактом и меньшей рутины;
> риск «грязного» контракта вынесен в обязательный ручной слой мапперов (см. выше).
> Переключение обратно на ручные DTO остаётся дешёвым (пакетов мало).

## MADR-008: Auth-слот — dev-заголовки сейчас, JWT (Telegram magic-link) потом

**Статус:** Proposed

**Контекст.** Backend сейчас без JWT (PoC, issues #85/#86): идентификация через `X-User-Id`
/ `X-Chat-Id`. Цель Sprint 1 — реальный `POST /care-events` уходит на backend. App Store
требует настоящий auth → Phase 3 вводит `/auth/telegram/*` + JWT (вариант Б), плюс Apple/Google.

**Решение.** Вводим абстракцию **`AuthSession`** (domain) и провайдер `authProvider`, за
которым прячется источник идентичности:
- **Sprint 1 (dev-слот):** `DevAuthSession` читает `CHAT_ID` / `USER_ID` из `--dart-define`
  и отдаёт их сетевому слою как scope-заголовки (MADR-006). Экраны 07/09 — заглушки,
  кнопка → `context.go('/home')`.
- **Phase 3 (JWT):** `JwtAuthSession` хранит access/refresh в `flutter_secure_storage`,
  отдаёт `Authorization: Bearer`. Меняется **только** реализация `AuthSession` и одна
  ветка в AuthHeaderInterceptor — экраны, репозитории и провайдеры не трогаются.

Контракт `AuthSession` (примерно): `headersFor(AuthScope) → Map<String,String>`,
`isAuthenticated`, `signOut()`. Интерсептор спрашивает у текущей сессии заголовки по scope.

**Причины.** Слот изолирует «грязный» PoC-auth за одним интерфейсом. Когда придёт JWT,
не будет «переписать всё» — будет «дописать вторую реализацию». Заголовки не размазаны по
data sources (см. MADR-006), а собраны в одном месте.

**Последствия.** `CHAT_ID`/`USER_ID` — только dev-flavor, **никогда** не в prod-бандле и
не в git (через `--dart-define`, см. MADR-010). prod-flavor без JWT просто не аутентифицируется —
это ок, prod-сборку не релизим до Phase 3.

## MADR-009: Локальное хранилище / offline — drift

**Статус:** Proposed

**Контекст.** Roadmap закладывает offline-sync (client-generated UUID + updated_at,
last-write-wins, как на backend). В Sprint 1 полноценный offline не нужен — нужен **каркас**
БД, чтобы потом не переезжать. Альтернатива — Isar (быстрее на больших данных, но другой
ментальный профиль; для backend-разработчика SQL ближе).

**Решение.** `drift` 2.33 (типобезопасный SQLite, реактивные запросы как `Stream`).

**ВАЖНО про нативную зависимость.** `sqlite3_flutter_libs` в реестре уже `0.6.0+eol`
(«Not used anymore, update to version 3.x of package:sqlite3»). drift 2.33 + Dart 3.12
используют **native assets** через `sqlite3` 3.x — отдельный `sqlite3_flutter_libs`
**не добавляем**. Это сверено на pub.dev 2026-05-27 и должно быть зафиксировано в каркасе,
иначе скопированный из старых гайдов `pubspec` потянет EOL-пакет.

**Что кешируем в Sprint 1: почти ничего.** Каркас БД (`AppDatabase`, пустые/минимальные
таблицы, `LocalDataSource`-провайдер) создаём, но фичи Sprint 1 ходят прямо в сеть через
Riverpod-кеш. Это сознательно: offline-стратегия — отдельная задача, преждевременный кеш на
PoC-контракте без `updated_at` в части DTO будет переписан.

**Стратегия sync (проектная, реализуется позже — НЕ в Sprint 1):**
- Каждая локальная запись care-event: `clientId (UUID, PK)`, `updatedAt`, `synced (bool)`.
- Запись ухода офлайн → пишем локально с `clientId`, ставим `synced=false`.
- При сети — отправляем `POST /care-events` с тем же `clientId` (идемпотентность backend
  гарантирует отсутствие дублей), по 201 ставим `synced=true`.
- Конфликты: **last-write-wins по `updatedAt`**, как на backend (никакого merge UI).
- Eventual consistency: UI читает локальную таблицу (Stream), фон синхронит.

**Причины.** drift = SQL-мышление (близко backend-разработчику), типобезопасные запросы,
реактивные стримы дружат с Riverpod, in-memory режим для тестов. Идемпотентность care-events
по `clientId` делает offline-sync почти бесплатным (не нужен сложный конфликт-резолвер).

**Последствия.** Каркас БД в `core/storage/`. Не тащить `sqlite3_flutter_libs`. Тесты
data-слоя — на in-memory drift. Реальное наполнение offline — issue после Sprint 2.

## MADR-010: Flavors / окружения — dev / prod

**Статус:** Proposed

**Контекст.** На backend есть профили; нужно то же на клиенте: разные base URL, bundle id,
dev-заголовки. Секреты (CHAT_ID/USER_ID сейчас, OAuth/RevenueCat позже) — не в git.

**Решение.** Два flavor: **dev** и **prod**, разные entrypoint'ы `main_dev.dart` /
`main_prod.dart`. Конфиг — через `--dart-define` (не `.env`-файл в репозитории):

```
# dev
flutter run --flavor dev -t lib/main_dev.dart \
  --dart-define=API_URL=https://plants-care.up.railway.app \
  --dart-define=CHAT_ID=9000001 --dart-define=USER_ID=1
# prod
flutter build ipa --flavor prod -t lib/main_prod.dart \
  --dart-define=API_URL=https://plants-care.up.railway.app
```

- bundle id: `com.<org>.plantcate.dev` / `com.<org>.plantcate` (раздельные → можно
  держать оба билда на одном устройстве).
- Разные иконки/имя приложения по flavor (видно, что dev).
- `--dart-define` значения локально — в `run-dev.sh` / VS Code launch.json (**в .gitignore**),
  в CI — из секретов.

**Причины.** `--dart-define` компилирует константы в бандл (нет рантайм-файла с секретами),
работает на CI, и dev/prod нельзя перепутать (разный entrypoint + bundle id).

**Последствия.** iOS — настройка Xcode schemes (dev/prod) + xcconfig. Android — productFlavors
в Gradle (Phase 3). `--flavor` без настроенных schemes падает — это часть каркаса.

## MADR-011: Error handling — Result/Either на freezed sealed

**Статус:** Proposed

**Контекст.** Нужен единый способ протаскивать ошибки backend (`{error:{code,...}}`) до UI,
чтобы экран мог показать осмысленное состояние (валидация по полю, «нет доступа», «оффлайн»).
Варианты: бросать исключения и ловить глобально, либо возвращать Result/Either.

**Решение.** Гибрид с явной границей:
- **data → domain → presentation: `Result<T>`** — freezed sealed `Result.success(T)` /
  `Result.failure(ApiError)`. Use case и репозиторий возвращают `Future<Result<T>>`, не
  бросают. `ApiError` — sealed из MADR-006.
- **Внутри data** (dio) исключения нормальны — ErrorInterceptor ловит `DioException`,
  репозиторий конвертит в `Result.failure`. Дальше исключения не «летят».
- **UI** — `AsyncValue` Riverpod уже даёт loading/error; для доменных ошибок Notifier
  кладёт `ApiError` в state и экран рисует нужный баннер/тост по типу ошибки.

Отдельный пакет под Either (dartz/fpdart) **не тянем** — freezed sealed + pattern matching
(Dart 3) закрывает потребность нативно.

**Причины.** Result делает ошибки видимыми в сигнатуре (компилятор заставляет обработать),
что ловит баги на этапе сборки — это и есть аргумент для новичка: «нельзя забыть про ошибку».
Трейд-офф — чуть больше `switch` в Notifier'ах; приемлемо.

**Последствия.** Контракт: domain/data **не бросают** наружу (бросок = отказ ревью).
Все доменные ошибки — через `ApiError`. UI маппит `ApiError` → текст пользователю
(тексты — через l10n, MADR-012).

## MADR-012: Локализация — ru-only сейчас (через AppLocalizations)

**Статус:** Accepted (2026-05-27, владельцем) · решение разошлось с рекомендацией архитектора

**Контекст.** Open question README §9 / SPRINT-1 §0.5: ru-only или ru+en сразу. Дизайн,
контент и `care-types` сейчас на русском. Backend отдаёт локализуемые тексты (`displayName`,
`description`) и в Phase 4 примет `locale` в `PATCH /me`.

**Решение.** **ru-only**, **один локаль (ru)**, но **инфраструктура l10n заводится с первого
дня**: `flutter_localizations` + `l10n.yaml` + `app_ru.arb`. **Все UI-строки идут через
сгенерированный `AppLocalizations`, не строковые литералы в виджетах.** Второй локаль (en)
не заводим, пока он реально не понадобится.

Это компромисс: экономим на наполнении второго ARB сейчас, но **не платим ретрофитом** —
добавить `app_en.arb` позже = создать файл с переводами, а не выковыривать строки из 24 экранов.

**Причины (выбор владельца).** Аудитория и контент на старте русскоязычные; второй язык —
не приоритет Sprint 1. Быстрее и меньше церемоний. Ключевое условие, снимающее главный риск
ретрофита, сохранено: **хардкод строк в виджетах запрещён даже при одном локале** (идёт через
`AppLocalizations`).

**Последствия.** `l10n.yaml` + `lib/core/l10n/arb/app_ru.arb` (единственный, полный).
`supportedLocales` — только `ru`. Тексты ошибок (MADR-011) — тоже в ARB. Voice lines растений
(бренд-фича, README §9.2) — если генерятся на клиенте, тоже через l10n по `taskType`. Добавление
`en` — отдельная задача (создать `app_en.arb` + расширить `supportedLocales`), код экранов не трогается.

> **Отвергнутая альтернатива (рекомендация архитектора): ru+en с первого дня.** Аргумент —
> разница в усилии минимальна, а полноценные два локаля сразу убирают вопрос. Владелец выбрал
> ru-only ради скорости Sprint 1; риск дорогого ретрофита нейтрализован обязательным
> прохождением строк через `AppLocalizations` уже сейчас.

## MADR-013: CI/CD — GitHub Actions сейчас, Codemagic к релизу

**Статус:** Accepted (2026-05-27, владельцем) · совпало с рекомендацией архитектора

**Контекст.** Open question SPRINT-1 §0.5: Codemagic vs GitHub Actions. Контекст — соло,
iOS-first, backend уже на GitHub (`antonkupreychik/plants-care`), бесплатные tier'ы важны.

**Решение. Двухэтапно:**
- **Sprint 1…до релиза: GitHub Actions.** Workflow на PR: `flutter analyze` + `flutter test`
  + (опц.) сборка debug. macOS-раннеры на free tier дают iOS-сборку; код и так на GitHub —
  один источник, PR-гейт (зелёный CI до ревью, MADR Git/PR) из коробки.
- **К первому TestFlight/релизу: добавить Codemagic** для iOS code signing, TestFlight и
  Google Play upload. Codemagic заточен под Flutter-релизы (управление сертификатами,
  provisioning, App Store Connect) — на GitHub Actions это настраивается заметно болезненнее
  (fastlane match, секреты сертификатов руками).

**Причины.** На этапе «собирается / тесты зелёные» GitHub Actions бесплатнее и ближе к уже
существующему репозиторию-соседу. Боль iOS-релиза (подпись, TestFlight) приходит позже и
там Codemagic экономит дни — но платить за неё сейчас, когда релиза нет, незачем.

**Последствия.** Sprint 1: `.github/workflows/flutter-ci.yml` (analyze + test). Codemagic —
отдельная задача перед Phase 3 (когда появится реальный auth и есть что выкладывать в TestFlight).
Лейбл `mobile` на issues/PR (как на backend).

## MADR-014: Тестирование

**Статус:** Proposed

**Контекст.** Нужна стратегия, выполнимая соло, и гейт «`flutter analyze` чисто +
`flutter test` зелёное» перед «готово» (FLUTTER.md, SPRINT-1 checklist).

**Решение.** Пирамида:
- **Unit (domain)** — use cases и мапперы (вкл. `taskType→type`), без Flutter. `flutter_test` + `mocktail`.
- **Unit (data)** — репозитории на моках dio (`mocktail`), мапперы DTO↔domain (включая ловушки
  «грязного» контракта), drift на **in-memory** БД. Сгенерированные DTO (MADR-007) не тестируем.
- **Notifier-тесты** — через `ProviderContainer` с `overrides` (подменяем репозиторий моком),
  проверяем переходы `AsyncValue`/`Result`.
- **Widget-тесты** — ключевые экраны (Home, Plant card, Water sheet) с overridden провайдерами.
- **integration_test** — критичные флоу: «открыть карточку → Полить → запись в дневнике» и
  (Phase 2) онбординг/добавление растения.

**Причины.** Domain/data-юниты быстрые и ловят логику (там и живут ловушки контракта).
Widget/integration — дороже, потому точечно на ключевых флоу, не на всех 24 экранах.

**Последствия.** Clock инжектится провайдером (FLUTTER.md «Время») → время в тестах
детерминировано. Каждый маппер DTO↔domain — тест. CI (MADR-013) гоняет `analyze` + `test` на PR.
